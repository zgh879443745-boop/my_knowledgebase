# LLM Wiki 和 GBrain 真正的差别2

## 原始内容

如果你现在还在给AI喂PDF、喂网页、喂笔记，然后期待它变聪明，那你可能搞错了一层。真正厉害的不是让AI多读点资料，而是开始有人在给AI造知识操作系统了。

这就是我看LLM Wiki和GBrain最大的感受。他们看起来都在做AI知识库，但实际上回答的是两个完全不同的问题：一个在回答知识怎么被编译，另一个在回答知识怎么持续运行。

### LLM Wiki 路线

这里有个点要先讲清楚：LLM Wiki这个思路最早来自Andrej Karpathy提出的LLM Wiki Pattern。它的核心想法不是把一堆资料直接丢给模型临时检索，而是先把原始资料整理成一套稳定的Wiki知识层，再让AI在这套知识层上工作。

后来社区开始把这套方法论做成具体项目，所以LLM Wiki更像是把Karpathy的"知识先编译再使用"这条路线产品化了。

### GBrain 路线

再看GBrain，这个项目的创始人就是GBrain的CEO，所以它不是一个纯演示性质的知识库Demo，而更像是一个长期在用Agent的人直接把自己需要的知识运行系统做出来了。

## 核心观点

他们共同反对的其实都是传统RAG，因为传统RAG本质上还是临时检索：
- 你把PDF往向量数据库扔进去
- 系统临时捞几段相关文本
- 再交给模型拼答案

它方便，但问题也很明显：
1. 同一个问题多问几次，答案可能不一样
2. 新资料进来了，旧知识是不会自动重组的
3. 问完就结束，下次还得重新捞材料

所以真正的问题不是AI能不能收到信息，而是AI能不能把零散资料慢慢沉淀成一套可复用、可维护、可持续更新的知识层。

### LLM Wiki的答案

LLM Wiki给出的答案是：**先把资料编译成百科，再让AI用它**。

它的核心单位是**页面**：
- 底层是原始资料
- 中间层是按主题、人物、概念整理好的Wiki页面
- 顶层是一套规则文件，规定知识怎么收录、怎么查询、怎么维护

所以它关心的重点不是收了多少资料，而是**有没有形成稳定的知识表达**。LLM Wiki更像一套知识编辑流程。

### GBrain的答案

而GBrain走的是另一条路。如果说LLM Wiki回答的是"知识怎么写"，那GBrain回答的是"知识系统怎么持续运行"。

它保留了页面这种可读可编辑的形态，但在上面又加了一整层工程基础设施：
- 数据库
- 向量检索
- 关键词索引
- 自动化流程
- 多媒体处理

所以GBrain不只是一个知识组织方法，它更像一个**长期运行的知识系统**。

#### 实际例子

比如一场客户会议录音：
- 在普通知识库里，它可能只是一个附件
- 但在GBrain，它会被转录、成页、更新实体、写入时间线，再建立索引
- 你后面提问时，面对的已经不是原始录音，而是一组被拆解和结构化的知识对象

## 两条路线的本质差别

- **LLM Wiki**：更像是在给AI写一本持续更新的百科全书
- **GBrain**：更像是在这本百科之上再搭一个知识操作系统

一个偏**知识表达层**，一个偏**知识运行层**。

## 如何选择？

- 如果你做的是**个人研究、写作、素材管理、小规模专题整理**，LLM Wiki会更轻、更透明
- 但如果你是给**Agent整达长期记忆**，要接多种信息源，还要支持会议、语音、视频这类多媒体输入，并且让知识自动更新，GBrain会更完整

## 思考

最最后留一个问题：你更认同哪条路线？是先把知识编译成Wiki，还是直接把知识做成一个持续运行的系统？

## 笔记

- LLM Wiki = 知识编译 + Wiki表达
- GBrain = 知识运行 + 工程基础设施
- 传统RAG的问题：临时检索，无持久化知识层
- 未来趋势：知识操作系统，而非简单知识库
